- Period: 2026.01.01 – 2026.03.27
- Type: Team Project / Autonomous Driving Competition
- My Role: Vision Perception & Local Path Generation
- Environment: ROS1 , Ubuntu 20.04 , OpenCV, Python

## 3. Problem & Solving

대회 특성상 1/5크기의 자동차에 노트북을 올려 주행했습니다.
노트북 특성상 차선 인지에 딥러닝을 사용하기에는 fps 와 지연성의 우려를 가졌습니다.
그래서 딥러닝 모델을 사용하지 않고 전통적인 OPENCV 방식으로 차선 구분을 시도하였습니다.

2. 차선 소실

강한 조명, 그림자, 급커브 또는 차선 훼손 상황에서는 일부 프레임에서
차선 픽셀이 부족해지고, 프레임 단위 검출만 사용하면 중심 경로가
즉시 사라지거나 크게 흔들리는 문제가 있었습니다.

#solution
이전 프레임의 차선 fitting 결과 주변을 우선 탐색하는
Search Around Polynomial 방식을 적용했습니다.

추적 실패 시에는 좌·우 영역을 분리한 Sliding Window 기반 전역 재탐색을
수행했습니다. 또한 최소 픽셀 수, 차선 폭, 좌우 역전 여부를 검증하여
노이즈 기반의 잘못된 차선 fitting을 제거했습니다.

짧은 차선 소실에는 이전 fit을 최대 2프레임 유지하고,
EMA smoothing을 적용하여 중심 경로의 시간적 연속성을 확보했습니다.
한쪽 차선만 검출되는 경우에는 고정 차선 오프셋을 이용해 중심 경로를
계속 생성했습니다.


좌우 구분

기존에는 단순히 가운데를 기준으로 왼쪽 차선 오른쪽 차선을 구분하였습니다.
급커브 이후 진입할때 차선인식이 반대로 되는 문제가 발생했습니다.

#solution

이전 프레임에서 생성된 중심 경로의 가장 가까운 하단점을 추출하고,
이를 다음 프레임의 동적 중심점으로 사용했습니다.

동적 중심점을 기준으로 좌·우 차선의 위치를 검증하여,
왼쪽 차선이 중심점 오른쪽으로 넘어가거나 오른쪽 차선이 중심점 왼쪽으로
넘어가는 비정상 fit을 제거했습니다. 또한 두 차선 간 폭이 비정상적으로
좁을 때, 어느 쪽에 차선이 몰렸는지 판단하여 잘못된 검출 결과를 제거했습니다.

이를 통해 차량의 진행 방향과 이전 경로를 반영한 기준점에서
차선 추적과 좌·우 차선 연관성을 유지하도록 구성했습니다.

## 4. System Architecture
Camera Image
→ Undistortion
→ Canny Edge Detection
→ IPM / Bird's-Eye View
→ Sliding Window Lane Tracking
→ Center Path Generation
→ Metric Coordinate Conversion
→ ROS nav_msgs/Path

## 5. Key Implementation

### BEV Transformation
카메라 외부 파라미터와 FOV를 이용하여 IPM 기반 BEV를 생성했습니다.
원근 왜곡을 줄여 차선의 기하학적 형태를 안정적으로 추정했습니다.

### Robust Lane Tracking
이전 프레임의 차선 fitting 결과 주변을 우선 탐색하고,
검출 실패 시 Sliding Window 기반 전역 재탐색을 수행했습니다.

### Single-Lane Path Estimation
한쪽 차선만 검출된 경우에도 이전에 추정한 차선 폭을 이용해
차로 중심 경로를 생성했습니다.

### ROS Path Publishing
BEV 영상의 픽셀 경로를 차량 기준 미터 좌표로 변환하고,
제어 모듈이 사용할 수 있도록 `nav_msgs/Path`로 발행했습니다.

## 7. Limitations

대회에서 비전구간에서는 전혀 문제가 일어나지 않았다. 차선인식을 수월하게 하였고 그에 맞는 경로를 생성했다.
전체 코드 구조에서의 한계로 인해 결과가 아쉽게 나왔다.
라이다 구간에서는 라이다 모드 비전 구간에서 비전 모드 이렇게 모드를 나눠서 진행을 했었다.
라이다 모드에서는 라이다에서의 정보만으로 주행을하고 비전 모드 에서는 차선 정보만을 이용해 주행을 하였다.
이 모드 전환의 트리거는 전방에 라이다 클러스터가 몇개 이상들어오면 라이다 모드로 변하는 것이었다.
이 모드 전환 과정에서 문제가 발생하였다. 차선이 없는 라바콘 구간에서도 비전은 켜져있어 잘못된 노이즈들을 차선으로 인식하고 그것을 물고 늘어졌다. 잘못된 결과를 물고있는데 그때 비전 모드로 전환이 이루어지면은 결과적으로 차선 좌우 구분을 못해 경로를 이탈하는 문제가 있었다. 그 외 차선 구간에서 라바콘이 들어오는 동적 장애물 구간에서 라이다 모드로 전환이 되어 멈춰야 하지만 그냥 회피를 하게 되는 문제가 있었다.

다시 생각해보면 모드를 나누는게 패착이었다. 그로 인해 범용성이 떨어졌고 문제가 발생한거다.
모드를 나누지말고 센서퓨전을 하여 라이다와 비전 정보를 한번에 보고 경로 생성을 했으면 문제가 발생하지 않았을 거다. 
또 다른 문제점은 카메라 너무 바닥으로 숙여 근거리만 보았던 문제이다. 측정했을때 카메라는 최대 5미터 앞까지 보는 상태 였다. 차의 속도가 빠르지 않고 최대 15km/h 인것을 생각해 충분히 제어 가능하다고 생각했다. 하지만 문제는 경로의 2m앞의 점을 추종하게 만들었다는 것이다.
그리고 대회 코스는 급격한 커브와 차선이 일반적인 커브가 아닌 직선으로 이루어져 있어 반응이 느렸다. 차의 속도에 따른 최적 경로 생성점을 동적으로 생성했으면 훨씬 잘되었을 것같다.
이것은 이후 논문을 참조해봐야하는 부분이다.

가장 큰 문제는 차량에 비해 너무 무거운 프로파일이었다. 카메라 gps 라이다 그리고 노트북을 거치하기 위해 프로파일을 만들었다. 처음에는 문제가 없었지만 날이 갈수록 차량이 가라앉으며 제어가 느려지며 마지막에는 방향전환 기어가 부서져 완주를 못하는 상황이 왔었다.
소프트웨어에 모든 정신이 팔려 하드웨어적 문제가 생길 것을 예상하지못했다. 
대회 직전에 문제는 알았지만 프로파일을 새로 구할 수 없는 상황이라 아쉬움이 더욱 컸다.
꼭 하드웨어적 성능을 평가해야한다고 다짐했다.
